# D-4. User / Admin Guide (요약)

## 1. 화면별 기능 설명 (예시)
- **Chat/Console**: 자연어 질의/명령 입력, Intent 결과/판단/BI 카드 표시, 피드백(👍/👎)
- **Workflow Builder**: 노드 카탈로그, DSL/JSON 뷰, 승인/버전 관리, 시뮬레이션 실행
- **BI Dashboard**: KPI 카드, 차트, 필터(라인/기간/제품), 분석 요약 텍스트
- **Monitoring**: 서비스 상태/알람, LLM/캐시/DB 지표, 로그 뷰어
- **Admin**: 사용자/RBAC, 커넥터/MCP 등록, 정책/Rule/Prompt 버전 관리, 설정

## 2. 주요 시나리오별 사용 절차
- **불량 알림 워크플로우 실행**
  1) Chat에 “L01 야간 불량 상태 알려줘” 입력
  2) 판단 결과/조치 카드 확인, 필요한 경우 승인/피드백
- **새 Rule 적용**
  1) Learning 탭에서 추천 Rule 후보 확인 → 시뮬레이션(Zwave) → 승인
  2) 적용 후 모니터링 대시보드에서 경보 변화 확인
- **BI 인사이트 조회**
  1) “지난주 불량 요약” 요청 → 차트/요약 확인
  2) 필터/기간 수정, CSV/이미지 내보내기(선택)
- **커넥터 등록**
  1) Admin → 커넥터 추가 → 엔진/DSN/인증 입력 → 테스트 → 저장
  2) ETL 스케줄 지정(일/시간) → 상태 확인
- **BI 카탈로그 관리**
  1) Admin → BI Catalog → 새 메트릭/차트 등록
  2) SQL/수식 정의 → 미리보기 → 저장 (이후 Chat에서 "OO차트 보여줘"로 호출 가능)

## 3. 관리자 설정/권한 관리
- 역할: admin/operator/viewer/approver 구분
- 정책: 워크플로우/Rule/Prompt/커넥터 변경은 승인자만 가능, 감사로그 자동 기록
- 인증: SSO/OIDC 연동(선택), API 토큰 관리, IP 제한(옵션)

## 4. FAQ / 트러블슈팅 (예시)
- 판단 응답이 느려요 → 캐시/LLM 상태 확인, 입력 스키마 검증, Rule-only 모드로 테스트
- JSON 파싱 실패 → 프롬프트 버전/템플릿 확인, 실패 로그에서 원인 파악 후 재시도
- MCP 호출 실패 → 서버/툴 등록 상태, 인증/네트워크, 회로차단 여부 확인
- 알람이 너무 많아요 → Hybrid Policy 임계 조정, Rule 재튜닝, Zwave로 영향 확인 후 조정
